# Chem-PDF-to-DB: Пайплайн извлечения SAR-данных из научных PDF

Данный репозиторий содержит пайплайн для автоматизированного извлечения данных о структуре и биологической активности химических соединений (SAR - Structure-Activity Relationship) из научных статей в формате PDF. 

Результатом работы пайплайна является единая нормализованная CSV-база данных (`data/database.csv`), объединяющая метаданные источников, химические дескрипторы, SMILES и числовые значения биологических измерений.

## Архитектура и методология

Пайплайн (`src/build_database.py`) состоит из 5 последовательных шагов:

### Шаг 1. Извлечение текста и метаданных из PDF
Библиотека `pdfplumber` используется для работы с PDF. Так как научные статьи имеют сложную верстку, реализованы несколько стратегий извлечения (`src/pdf_text.py`):
* **Двухколоночный режим (`full_text_by_column`):** Страница разрезается пополам (по оси X), текст левой и правой колонки извлекается отдельно, чтобы избежать склеивания строк из разных колонок.
* **Пространственное извлечение (`extract_rows_by_position`):** Для «плавающих» таблиц, которые могут пересекаться с основным текстом, слова извлекаются по их реальным координатам (bbox), кластеризуются по оси Y (с допуском) и сортируются по оси X. Это позволяет восстановить строки таблицы именно в том виде, в котором они расположены на странице.
* **Парсинг метаданных:** С первой страницы извлекаются: Журнал, Год, Том, Страницы, DOI и Название статьи.

### Шаг 2. Индивидуальный парсинг таблиц (`src/parsers/`)
Поскольку формат таблиц в статьях отличается, для каждой статьи написан свой кастомный парсер. 
* Парсер находит заголовок таблицы (например, `Table 1`) с помощью `extract_table_block`.
* Применяет специфичную для статьи логику: регулярные выражения для поиска ID соединений, маппинг колонок на конкретные рецепторы-мишени (KOR, MOR, DOR) и типы анализов (binding affinity, functional potency).

### Шаг 3. Очистка и стандартизация данных
Сырые данные приводятся к единому виду:
* **Стандартизация мишеней:** Словарь `TARGET_STD_MAP` приводит синонимы к единому виду (например, `MOP` -> `MOR`, `ORL1` -> `NOP`).
* **Парсинг значений:** Алгоритм корректно обрабатывает:
  * Цензурированные данные: `>1000` (censor: `above`), `<5` (censor: `below`).
  * Специфичные статусы: `nd` (not determined), `na` (not active).
  * Очистку от ошибок OCR/верстки: удаление знаков `±SEM` (стандартная ошибка) перед парсингом float-значения.

### Шаг 4. Дедупликация и разрешение конфликтов
Пайплайн может автоматически находить повторяющиеся измерения:
* **Точные дубликаты:** Если значения совпадают, они сливаются в одну запись (в `table_ref` и `note` сохраняется информация о том, что данные дублировались в нескольких местах).
* **Конфликты:** Если для одного соединения в разных местах указаны *разные* значения, запись не удаляется. Обе записи сохраняются, а в поле `conflict_flag` проставляется пометка `CONFLICT` с указанием ID конфликтующих строк для ручной проверки.

### Шаг 5. Обогащение SMILES-нотациями (`src/enrich_smiles.py`)
Для возможности дальнейшего молекулярного моделирования пайплайн пытается получить Canonical SMILES для каждого соединения:
* Формируется словарь поисковых запросов на основе химического названия (`descriptor`) или известных тривиальных имен (например, `naltrexone`).
* Осуществляются запросы к **PubChem PUG REST API**.
* Реализовано **кэширование** (`.smiles_cache.json`), чтобы не делать повторные запросы при перезапуске пайплайна.
* Соблюдается **Rate Limit** (0.34 сек между запросами) для соблюдения политик PubChem.
* Если название не найдено (например, в статье это просто "Compound 1a" без химического названия), поле `smiles` остается пустым, а в `smiles_note` пишется причина.

## Схема итоговой БД (`data/database.csv`)

Итоговая таблица содержит следующие поля:

**Метаданные соединения и источника:**
* `measurement_id` — уникальный ID строки.
* `compound_id`, `compound_label`, `descriptor` — идентификаторы и химическое описание из статьи.
* `smiles`, `smiles_source`, `smiles_note` — полученные SMILES, источник (PubChem) и статус поиска.
* `source_short`, `source_journal`, `source_year`, `source_doi`, `source_title` — библиографические данные статьи.

**Биологические данные:**
* `target_raw`, `target_std` — мишень (до и после стандартизации).
* `assay_category`, `assay_subtype` — категория и тип анализа (например, *binding_affinity* / *Ki (radioligand binding)*).
* `value_raw`, `value_std`, `unit_std` — исходное и нормализованное числовое значение.
* `censor` — статус цензурирования (`above`, `below`, `not_determined`).
* `table_ref` — ссылка на таблицу в PDF.
* `note`, `conflict_flag` — служебные пометки о дубликатах, границах значений и конфликтах.

## Известные ограничения

1. **Отсутствие OCR-модуля:** В статьях `Pettersson2009_JMC` и `Guerrini2009_JMC` таблицы с данными сохранены в PDF как **растровые изображения**, а не как текст. Парсер (OCR) для них в рамках данного пайплайна не реализован, поэтому данные из этих статей отсутствуют в итоговой БД.
2. **Невозможность извлечения SMILES для R-групп:** Если в статье соединение описано только как номер (например, `1a`) или общая формула с R-группами без конкретного химического названия, PubChem API не сможет найти SMILES. В таких случаях `smiles` остается пустым.
3. **Зависимость от сети:** Шаг обогащения SMILES требует доступа к интернету для первого прогона. При отсутствии сети пайплайн не упадет, но SMILES не будут найдены (что будет отражено в `smiles_note`).

## Запуск и требования

### Зависимости
```bash
pip install pdfplumber requests
```
*(Примечание: `requests` не является строго обязательным для базового парсинга, но необходим для шага обогащения SMILES. Если его нет, пайплайн корректно пропустит этот шаг).*

### Структура проекта
```text
.
├── data/                   # Папка с исходными PDF и итоговым database.csv
├── src/
│   ├── build_database.py   # Главный оркестратор пайплайна
│   ├── pdf_text.py         # Утилиты для работы с PDF (pdfplumber)
│   ├── sources.py          # Реестр статей и извлечение метаданных
│   ├── enrich_smiles.py    # Модуль запросов к PubChem API
│   └── parsers/            # Директория с кастомными парсерами для каждой статьи
│       ├── dolle2009.py
│       ├── yang2009.py
│       └── ...
└── README.md
```

### Как запустить
Для запуска полного цикла извлечения и сборки БД выполните:
```bash
python src/build_database.py
```
Результат будет сохранен в `data/database.csv`.
